# Telegram Bridge Fallback

This plugin provides a minimal executable Telegram bridge with two flows:
- inbound Telegram webhook routing into the CEO route.
- outbound Telegram notifications for Paperclip issue events.

## Root Cause (FAV-27)

The workspace only contained a documentation scaffold and no executable routing implementation. Incoming Telegram updates had no runtime path to enter Paperclip routing.

## Fix Summary

- Added executable bridge handler with:
- `POST`-only webhook handling.
- Webhook path validation via `TELEGRAM_WEBHOOK_PATH`.
- Built-in CEO intake endpoint `POST /paperclip/telegram/ceo`.
- Telegram-native webhook validation via `X-Telegram-Bot-Api-Secret-Token` and
  `TELEGRAM_WEBHOOK_SECRET_TOKEN` (fallback to `Authorization: Bearer TELEGRAM_BOT_TOKEN`
  when no webhook secret token is configured).
- Optional chat allow-list gate via `TELEGRAM_ALLOWED_CHAT_ID`.
- Telegram update normalization (`message` + `callback_query`) into a single internal event shape.
- Explicit forwarding into CEO route handler.
- Configurable forwarding from CEO route handler into Paperclip endpoint via
  `PAPERCLIP_CEO_FORWARD_URL` (+ optional bearer token/timeout).
- Added issue-event notifier handler with:
- `POST`-only issue event endpoint via `PAPERCLIP_ISSUE_EVENT_PATH`.
- Authorization validation against `PAPERCLIP_WEBHOOK_TOKEN`.
- Issue payload formatting into Telegram text message.
- Telegram `sendMessage` delivery using `TELEGRAM_BOT_TOKEN` and `TELEGRAM_NOTIFY_CHAT_ID`.
- Optional status filter via `TELEGRAM_NOTIFY_STATUSES`.
- Transition gate-lock for status changes to protected states (`in_review,done` by default) requiring Reality/Proof completion flags in payload (`realityGateComplete`, `proofGateComplete`, or equivalent nested gate fields).
- Retry/backoff for transient Telegram API failures via `TELEGRAM_SEND_MAX_RETRIES` and `TELEGRAM_SEND_RETRY_BACKOFF_MS`.
- Optional host allow-list gate via `PAPERCLIP_ALLOWED_HOSTNAMES` using `Host`/`X-Forwarded-Host`.
- Added minimal Node HTTP server entrypoint for local runtime smoke.
- Added E2E smoke tests for both CEO routing and issue-event notification delivery.

## Required Secrets And Headers

`PAPERCLIP_WEBHOOK_TOKEN` is a shared secret you define yourself. It is not
generated by Telegram and not a fixed Paperclip constant.

Set it in the runtime environment of this server process:

```bash
export PAPERCLIP_WEBHOOK_TOKEN="replace-with-long-random-secret"
```

Every caller of the issue-event endpoint must send the same value:

```http
Authorization: Bearer <PAPERCLIP_WEBHOOK_TOKEN>
```

For inbound Telegram webhooks, prefer Telegram's secret-token mechanism:

```bash
export TELEGRAM_WEBHOOK_SECRET_TOKEN="replace-with-long-random-secret"
```

Configure Telegram webhook with that same secret token. The fallback bridge
validates `X-Telegram-Bot-Api-Secret-Token` against
`TELEGRAM_WEBHOOK_SECRET_TOKEN`.

Backward compatibility remains available: if
`TELEGRAM_WEBHOOK_SECRET_TOKEN` is unset, the bridge falls back to checking
`Authorization: Bearer <TELEGRAM_BOT_TOKEN>`.

## Known Errors

- `missing_bot_token`:
  `TELEGRAM_BOT_TOKEN` is missing in runtime env.
- `missing_webhook_token`:
  `PAPERCLIP_WEBHOOK_TOKEN` is missing in runtime env.
- `unauthorized`:
  Incoming `Authorization` header token does not match configured secret.
- `not_found`:
  Request path does not match `TELEGRAM_WEBHOOK_PATH`, `/paperclip/telegram/ceo`, or
  `PAPERCLIP_ISSUE_EVENT_PATH`.
- `host_not_allowed`:
  Request host did not match `PAPERCLIP_ALLOWED_HOSTNAMES`.
- `transition_blocked_missing_reality_proof_gates`:
  Protected status transition was blocked because Reality/Proof gates were not marked complete.
